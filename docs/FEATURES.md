## Auth

| Status | Feature | Where |
|---|---|---|
| ✅ | `POST /api/auth/login` — username + password, returns a short-lived `challengeId` (not yet logged in). Case-insensitive username. Uniform timing regardless of whether the user exists (always runs bcrypt) | `server/src/auth/routes.ts` |
| ✅ | `POST /api/auth/verify-totp` — six-digit TOTP with ±1 step tolerance (accounts for clock skew). On success, issues a JWT (HS256, 30-day TTL) in an httpOnly session cookie | same |
| ✅ | Peek-then-consume: a wrong TOTP **does not** burn the challenge — the user can retry without going back to the password screen | same |
| ✅ | **2FA recovery codes** — 10 single-use fallbacks issued at `claudex init` (printed once to stdout) and regeneratable from Settings → Security. Code shape `xxxx-xxxx-xxxx-xxxx`, 16 alphanumeric chars from a confusion-free alphabet (no `0/1/o/l/i`), bcrypt-hashed at rounds=10. `POST /api/auth/verify-recovery-code` (body `{challengeId, code}`) mirrors the TOTP flow: shares the per-challenge sliding-window limiter, peek-then-consume on the challenge, flips `used_at` on match, and issues the same session cookie. Response body also reports `remaining` so UIs can nag. `POST /api/auth/recovery-codes/regenerate` (login-gated) wipes the prior batch (used or not) and returns the 10 plaintext codes exactly once. `GET /api/auth/recovery-codes/state` (login-gated) returns `{remaining, generatedAt?}` — never plaintext. Audit events: `recovery_codes_regenerated`, `recovery_code_used`, `recovery_code_failed`. Stored in migration id=12 `recovery_codes(user_id, code_hash, used_at, created_at)` with index on `(user_id, used_at)` | `server/src/auth/recovery-codes.ts`, `server/src/auth/routes.ts`, `server/src/auth/index.ts` (`UserStore.setRecoveryCodeHashes / consumeRecoveryCode / countRemainingRecoveryCodes`), `server/src/bin/init.ts`, `server/src/db/index.ts` (migration 12), `web/src/screens/Settings.tsx` (`RecoveryCodesCard`, `RecoveryCodesModal`), `web/src/screens/Login.tsx` (`RecoveryCodeInput` toggle) |
| ✅ | Cookie `Secure` attribute auto-detected from request protocol and `X-Forwarded-Proto`; override with `CLAUDEX_COOKIE_SECURE=1/0`. `SameSite=Lax`, `HttpOnly`, `Path=/` | same |
| ✅ | `GET /api/auth/whoami` — returns the logged-in user or 401 | same |
| ✅ | `POST /api/auth/logout` — clears the session cookie | same |
| ✅ | `POST /api/auth/change-password` — logged-in only. Requires the caller's current password (401 `invalid_credentials` on mismatch), min-8 on new password (400 `bad_request`), rejects identical-to-current (400 `same_password`). On success rotates the bcrypt hash and issues a fresh session cookie so the caller's tab keeps working. The JWT secret is not rotated, so other tabs' old cookies remain valid until their own `exp` — we don't maintain a revocation list yet | `server/src/auth/routes.ts`, `server/src/auth/index.ts` (`UserStore.setPasswordHash`) |
| ✅ | `GET /api/user/env` — logged-in only. | ✅ | JWT signed with a 48-byte random secret persisted at `~/.claudex/jwt.secret` (mode 0600) | `server/src/auth/index.ts` |
| ✅ | Bcrypt password hashing (12 rounds); rejects passwords under 8 chars | same |
| ✅ | `otplib` TOTP with issuer `claudex`; current-code helper exposed for tests | same |
| ✅ | Short-lived login challenge store (5 min TTL) with peek + consume semantics | same |
| ✅ | **Rate limiting** on the auth endpoints — in-memory sliding-window limiter (`SlidingWindowLimiter`). `POST /api/auth/login` keyed by `req.ip`, max **5 failed attempts per 5 min**; `POST /api/auth/verify-totp` keyed by `challengeId`, max **10 failed attempts per 15 min** (stricter because the TOTP space is only 10^6). Successful credentials reset the login counter; a successful TOTP resets the challenge counter. Over-limit responses are `429 {"error":"rate_limited","retryAfterSec":N}` with a `Retry-After` header. Rate-limit trips audit as `login_rate_limited` / `totp_rate_limited` alongside the existing `login_failed` / `totp_failed` rows | `server/src/auth/rate-limit.ts`, `server/src/auth/routes.ts` |

Disabled under `NODE_ENV=test` (tests drive their own cases) and `CLAUDEX_WATCH_CLI=0` (emergency off-switch). Graceful shutdown awaits `watcher.close()` before DB close. **Honest limitations**: read-only mirroring — we only observe "file grew recently" / "file stopped growing", not "CLI process is alive"; paused-but-background runs may briefly flip idle → running on next append. Assumes default CLI layout `~/.claude/projects/<slug>/<uuid>.jsonl`; users with a relocated CLI state dir won't be observed. JSONLs at unexpected depths (not `<root>/<slug>/<file>`) are ignored for safety. **Known issue (pre-fix data)**: users who ran claudex between the watcher landing (`c455566`) and this fix may have duplicate `session_events` rows — each claudex-driven turn was imported a second time from the SDK's JSONL echo. Manual cleanup is recommended on a per-session basis: eyeball `SELECT seq, kind, substr(payload,1,80) FROM session_events WHERE session_id='…' ORDER BY seq;` for adjacent-seq pairs with identical kind+payload, then `DELETE FROM session_events WHERE id IN (…)` the confirmed duplicates. ## How to keep this file honest

- Every commit that adds a user-visible behavior, a new API surface, or moves
  an item between the three tiers **must** update the corresponding row in this
  file in the same commit.
- Every commit that deletes a feature must remove (or downgrade) its row.
- If you introduce a capability on the backend but intentionally don't expose
  it yet, add it as 🟡 with a one-line note on what's missing. That way nobody
  re-implements it or misses the last 10%.
- Don't reorganize the doc without a reason — stable anchors let humans and
  agents reference specific rows in review.
